# Documentation Map

This folder contains both operational docs and reference docs.

The repository is a harness first. Read documents in the right order so you do
not confuse input material, accepted truth, work packet, proof, and history.

## Read Order For New Sessions

1. `README.md`
2. `docs/HARNESS.md`
3. `docs/HARNESS_LIFECYCLE.md`
4. `docs/FEATURE_INTAKE.md`
5. relevant product docs under `docs/product/`, if they exist
6. relevant stories under `docs/stories/`
7. `docs/TEST_MATRIX.md`
8. relevant decisions under `docs/decisions/`
9. support docs such as `docs/VERSIONING.md`, `docs/TRACEABILITY.md`, and `docs/GLOSSARY.md`

## Operational Docs

These files define how work should be routed and reviewed.

- `docs/HARNESS.md`
- `docs/HARNESS_LIFECYCLE.md`
- `docs/FEATURE_INTAKE.md`
- `docs/TEST_MATRIX.md`
- `docs/VERSIONING.md`
- `docs/TRACEABILITY.md`
- `docs/HARNESS_BACKLOG.md`

## Product Truth

`docs/product/` holds accepted product behavior once a real project spec is
converted into living docs.

## Work Packets

`docs/stories/` holds bounded story contracts and high-risk packets.

## Historical Reasoning

`docs/decisions/` holds durable rationale for meaningful choices.

## Templates

`docs/templates/` holds reusable authoring structures. Templates should remain
generic enough for future real projects, but concrete enough to reduce guesswork.

## Support Docs

- `docs/GLOSSARY.md` normalizes terms
- `docs/README.md` maps the system
- `docs/VERSIONING.md` controls version and changelog behavior
- `docs/TRACEABILITY.md` controls artifact linking expectations
